# Board definitions

YAML files here are the **only** source of truth for kit boards — one file per board (`<id>.yaml`). CI and `generate-manifest.py` discover them automatically.

Builds point **`ESPD_BOARDS_DIR`** at this directory (the `espd` submodule is not modified and board YAMLs are not copied into it).

Schema and authoring guide: **[espd/docs/ADDING_A_BOARD.md](../espd/docs/ADDING_A_BOARD.md)** (in submodule).

`build-board.sh` and CI write **`CONFIG_ESPD_BOARD_<ID>=y`** to **`espd/sdkconfig.defaults.local`** from the YAML `id`.

## BSP profiles (`boards/profiles/`)

Optional **`boards/profiles/<bsp.component>.yaml`** files hold shared tuning for
all kits using the same esp-bsp component (UART console, `sdkconfig:` map,
`lcd_types_off:` list). `gen_board_plugins.py` merges them at configure time;
per-board `bsp:` keys override the profile.

## Flasher metadata

The web flasher reads board entries from release `manifest.json` (generated by `scripts/generate-manifest.py`):

| Field | Role |
|-------|------|
| `name` | Board title in the flasher list |
| `note` | Optional subtitle (second line) — put next to `name` at the top of the YAML |
| *(auto)* `image` | `flasher/assets/boards/<id>.jpg` when that file exists (400 px max edge); omitted otherwise |

`help` stays for menuconfig and docs. Optional `flasher.image` in YAML overrides the auto path (e.g. shared hardware photo before you add a symlink).

Board photos live in `flasher/assets/boards/`. Variants that share a product shot can symlink (e.g. `waveshare_s3_ap.jpg` → `waveshare_s3.jpg`).

## Adding a board

1. Add `boards/<id>.yaml`
2. Tag a release — CI builds firmware and attaches `manifest.json`; Pages mirrors `flasher/manifests/releases/{tag}.json`
